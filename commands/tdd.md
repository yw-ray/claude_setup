# `tdd` - Test Driven으로 작업 개시

## 사용법
- `/tdd` 또는 `/tdd <작업 설명>`

## 목적
이 명령을 사용하면 **Test Driven Development (TDD)** 방식으로 작업을 시작합니다. 구현보다 테스트를 먼저 작성하고, Red → Green → Refactor 사이클을 따릅니다.

## 동작

1. **작업 범위 확인**
   - `/tdd`만 입력한 경우: **진행 중인 맥락**이 있으면 그 작업으로 TDD를 진행합니다. 맥락 예: 최근에 논의한 플랜 문서(.claude/plans/), 최근 대화에서 정한 다음 작업(예: “다음 할 첫 작업”), 열린 플랜의 작업 범위 등. 맥락이 없을 때만 구현할 기능·수정 대상을 사용자에게 묻습니다.
   - `/tdd <작업 설명>`인 경우: 해당 설명을 기준으로 TDD로 진행할 작업을 정리합니다.

2. **TDD 사이클 적용**
   - **Red**: 요구사항에 맞는 실패하는 테스트를 먼저 작성합니다.
   - **Green**: 테스트를 통과하는 최소 구현을 작성합니다.
   - **Refactor**: 필요 시 리팩토링 후에도 테스트가 통과하는지 확인합니다.

3. **진행 순서**
   - 테스트 코드 작성 → 테스트 실행(실패 확인) → 구현 → 테스트 실행(통과 확인) → 리팩토링

## AI Assistant 가이드

- 이 명령이 호출되면, 구현 제안 전에 **먼저 테스트 작성**을 제안하고 수행합니다.
- 테스트 파일/케이스와 구현 대상(모듈·함수)을 명시하고, 테스트 → 구현 순서를 유지합니다.
- 테스트 실행 명령(예: `pytest`, `manage.py test`)을 함께 안내합니다.
- 프로젝트별 테스트 규칙이 있으면 따릅니다 (Django: `TestBaseClass`, test-runner: `TestCaseBaseClass` 등).
