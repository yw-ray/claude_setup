# 개인 개발 워크플로우 (로컬 리포 격리)

여러 에이전트(또는 Cursor 세션)가 **코드 작업을 서로 간섭 없이** 실행되도록 하는 워크플로우입니다. 핵심은 **에이전트마다 별도 로컬 리포(클론)**를 두는 것입니다.

## 목표

- 에이전트 A의 편집·커밋이 에이전트 B의 워크스페이스를 덮어쓰지 않음
- 각 에이전트는 자신만의 워크트리에서 `new`, `fork`, `switch` 등을 사용

## 워크플로우 요약

1. **플랜 먼저**: 대체로 플랜(작업 범위)을 먼저 정한 뒤 `clone` 실행.
2. **격리된 워크스페이스 생성**: `clone` 또는 `clone <suffix>` 로 현재 리포의 격리용 클론을 하나 더 만듦.
   - **접미사 생략 시**: 작업 내역(플랜 제목, 현재 브랜치, 최근 작업)을 보고 경로를 적당히 네이밍 (예: `mats-monorepo-feature-testconfig-api`). `clone` 커맨드가 제안하고 사용자 동의 후 진행.
   - **접미사 지정 시**: `clone <suffix>` 로 경로 생성 (예: `clone redis-role-separation` → `~/work/mats-monorepo-redis-role-separation`).
3. **새 Cursor 창에서 열기**: `clone` 실행 시 **`cursor <경로>`** 가 자동으로 실행되어 해당 경로가 새 Cursor 창에서 열린다. 이미 있는 경로로 이동할 때는 `use <path>` 또는 `use <hint>` 로 같은 방식으로 연다.
4. **해당 창에서만 작업**: `new`, `fork`, `switch`, `sync` 등은 **현재 열린 워크스페이스(클론)** 안에서만 동작. 에이전트마다 다른 클론을 열어 두고, 각 클론에서 작업하면 코드 작업이 독립적으로 이뤄짐.
5. **변경 반영이 필요할 때**: 한 클론에서 만든 변경을 다른 클론이나 메인 리포에 반영하려면, 커밋 후 push/pull, cherry-pick, merge 등으로 처리.

## 커맨드와의 관계

| 커맨드 | 역할 | 로컬 리포 격리 시 |
|--------|------|-------------------|
| `clone` | 격리용 클론 생성 | 에이전트마다 한 번씩 실행해 별도 워크스페이스 확보. 생성 후 **`cursor <경로>`** 로 새 창이 열림. `clone 195` 처럼 PR 번호를 주면 해당 PR 브랜치를 체크아웃한 클론 생성 |
| `use` | 이미 있는 clone(또는 리포) 경로를 Cursor로 열기 | **`cursor <경로>`** 로 해당 폴더를 새 창에서 연다. 선언 없이 완전 분리. |
| `new` | upstream/main 기준 새 브랜치 생성 | **현재 열린 클론** 안에서 실행. 여러 에이전트를 쓸 때는 먼저 `clone`으로 클론을 나눈 뒤, 각 클론(열린 창)에서 `new` |
| `fork` | 현재(또는 지정) 브랜치에서 분기 | **현재 열린 클론** 안에서 실행 |
| `switch` | 브랜치/PR로 이동 | **현재 열린 클론** 안에서 실행 |
| `sync` | 특정 브랜치에 rebase | **현재 열린 클론** 안에서 실행 |

## 정리

- **한 에이전트만 쓸 때**: 하나의 리포만 열어 두고 `new`/`fork`/`switch` 사용하면 됨. `clone`은 선택.
- **여러 에이전트를 쓸 때**: `clone`으로 에이전트 수만큼 격리된 클론을 만들면 `cursor <경로>` 로 각각 새 창이 열리므로, 각 창에서 `new`/`fork`/`switch` 등을 사용하면 코드 작업이 독립적으로 실행됨. 이미 있는 클론으로 이동할 때는 `use <path>` 로 해당 경로를 Cursor에서 연다.
